Chain-aware pruning

Before deleting a decayed memory, the decay job checks its graph neighbours. If any neighbour is still strong, the memory is kept alive. Related facts age together — no isolated deletions that break a cluster.

node [3] strength = 0.03 < threshold
→ neighbour [1] strength = 0.90 ≥ threshold
→ chain_safe_to_prune = False — kept alive

Recall propagation

Recalling memory A automatically bumps the recall count of A's graph neighbours, weighted by edge strength. Frequently-used memories keep their entire context cluster fresh through the decay formula.

recall("Python backend")
→ [1] Python/MongoDB recalled
→ [2] DuckDB/spaCy recall_count++ (w=0.34)
→ [5] Docker/K8s recall_count++ (w=0.29)

Multi-agent shared memory.

Multiple AI agents share context or keep secrets. Each agent authenticates with an API key. You control exactly what each agent can read and write.

1

Register an agent

Each agent gets a unique ym_ API key. Shown once, hashed with SHA-256 — never stored in plaintext.

from src.services.api_keys import register_agent

result = register_agent(
  agent_id="coding-agent",
  user_id="sachit",
  can_read=["shared", "private"],
)
# → ym_xxxx (save once)
2

Store shared or private

Pass the API key in any MCP call. Set visibility to control who can see it.

# shared — all agents see this
store_memory(
  content="DB is Postgres 16",
  api_key="ym_xxxx",
  visibility="shared"
)

# private — only this agent
store_memory(
  content="staging key sk-xxx",
  visibility="private"
)
3

Recall with scope

Without a key → shared memories only. With a key → shared + that agent's private memories.

# coding-agent recalls
recall_memory(
  query="database production",
  api_key="ym_xxxx"
)
# ← shared + private memories

# review-agent (different key)
recall_memory(query="database")
# ← shared memories only

Visibility matrix

private ✓ ✗ ✗

Keys hashed with SHA-256 before storage. Revoke anytime with revoke_agent(agent_id, user_id).

Memory rules for your agent.

Drop MEMORY_RULES.md into your CLAUDE.md or system prompt to tell any agent exactly when and how to use memory.

1

Recall before acting

At the start of every task, retrieve memories relevant to the request. Use the context to answer without asking the user to repeat themselves.

recall_memory(
  query="auth token expiry",
  user_id="sachit"
)
2

Decide what to store

For every fact that surfaces, apply the store / update / ignore policy. Call the MCP tool immediately — do not batch or defer.

# new fact → store
store_memory(
  content="JWT expiry is 2h",
  importance=0.8
)
3

Update, don't duplicate

If new info extends or contradicts an existing memory, merge it into one sentence and call update_memory with the recalled ID.

# stale fact → update
update_memory(
  memory_id=42,
  new_content="JWT now 4h",
  importance=0.8
)

Store / update / ignore?

Case Condition Action
New No existing memory covers it store_memory
Extension Adds detail to a recalled memory update_memory
Conflict Contradicts a recalled memory update_memory
Ignore Question, filler, no lasting value —

A fact has lasting value if it reveals a preference, constraint, architectural decision, failure mode, or strategy. Never store Claude's own responses.

Importance scale

0.9–1.0

Core identity

"User's name is Sachit" · "Project uses PostgreSQL" · permanent facts

0.7–0.8

Strong preferences

Recurring patterns · architectural decisions · "never add docstrings"

0.5

Regular facts

Project choices · "decided to use DuckDB for local dev"

0.2–0.3

Transient context

Session-specific notes · "currently debugging the webhook"

Importance is required on every store and update call — the agent must decide it, never omit it.

Category controls decay rate

Pick the category that best matches the content. Ebbinghaus decay runs independently per category — failures fade fast, strategies persist longest.

fact ~24 days

User preferences, identity, stable knowledge. Default category.

strategy ~38 days

Approaches that worked well. Decays slowest — keep good strategies longest.

assumption ~19 days

Inferred beliefs, uncertain context. Fades before facts.

failure ~11 days

What went wrong. Stale errors fade fastest — no outdated noise.

strategy example: "Pagination with cursor solved the timeout on large DB queries" failure example: "OAuth failed for client X due to wrong redirect URI — fixed by adding /callback"

PostgreSQL (optional — teams / large datasets)

Install with Postgres support
$ pip install yourmemory[postgres]
Create a .env file
DATABASE_URL=postgresql://YOUR_USER@localhost:5432/yourmemory

Backend selected automatically — postgresql:// → Postgres + pgvector, anything else → DuckDB.

Neo4j graph backend (opt-in — scale)
$ pip install 'yourmemory[neo4j]'
$ GRAPH_BACKEND=neo4j yourmemory

Default graph backend is NetworkX (zero setup). Switch to Neo4j for production scale via GRAPH_BACKEND=neo4j.
